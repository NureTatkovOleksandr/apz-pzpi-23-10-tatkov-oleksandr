3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Загальні відомості про систему

Назва системи: Telegram (офіційна назва — Telegram Messenger).
Розробник: Компанія Telegram Messenger LLP (заснована в 2013 році Павлом Дуровим та Миколою Дуровим). Клієнтські додатки частково відкриті (наприклад, бібліотека TDLib), проте серверна частина є закритою та пропрієтарною. Розробка протоколу MTProto також належить команді Telegram (основний автор — Микола Дуров).
Призначення системи (область застосування, користувачі, основні функції): Telegram — це хмарна платформа миттєвого обміну повідомленнями, яка забезпечує швидку, безпечну та масштабовану комунікацію в глобальному масштабі. Область застосування охоплює особисте спілкування, бізнес-комунікацію, масові розсилки, спільноти та автоматизацію.
Користувачі: понад 900 мільйонів активних користувачів щомісяця (станом на 2025–2026 роки) — від приватних осіб і сімей до великих корпорацій, урядів, ЗМІ, неурядових організацій та розробників. Підтримуються десятки платформ: iOS, Android, Windows, macOS, Linux, веб-версія та навіть легкі клієнти.
Основні функції:
• Миттєвий обмін текстовими, голосовими, відеоповідомленнями, фото, відео та файлами до 2 ГБ.
• Голосові та відеодзвінки (включаючи групові).
• Групи до 200 000 учасників, канали з необмеженою кількістю підписників.
• Боти та Bot API для автоматизації (платежі, вебхуки, інлайн-режим).
• Секретні чати з наскрізним шифруванням (end-to-end).
• Хмарне зберігання повідомлень і медіа з синхронізацією між пристроями.
• Пошук, самоуничтожувані повідомлення, папки, теми, реакції, опитування, відеочати тощо.
• Інтеграція з ШІ-сервісами (боти на базі Grok, GPT тощо через Bot API).
Система орієнтована на швидкість навіть у слабких мережах, конфіденційність (сервер не зберігає вміст секретних чатів) та масштабованість без втрати продуктивності.

3.2 Архітектурний стиль системи

Telegram реалізовано переважно як клієнт-серверну архітектуру з елементами розподіленої мікросервісної архітектури та подієво-орієнтованої обробки (event-driven). Основний протокол — власний MTProto 2.0 (Mobile Protocol), який поєднує високорівневий API, криптографічний шар і транспортний шар.
Це не класична шарова архітектура (layered), а гібридна:
• Клієнт-серверна з централізованими дата-центрами (multi-DC).
• Мікросервісна на серверній стороні (Message Router, Group/Channel Engine, Media Processing, Push Service тощо).
• Подієво-орієнтована — повідомлення обробляються асинхронно через роутери та push-систему.
• Модульна на клієнтській стороні (TDLib як ядро для всіх платформ).
Обґрунтування вибору архітектури: MTProto розроблений спеціально для мобільних пристроїв і ненадійних мереж (мобільний інтернет, Wi-Fi). Він забезпечує мінімальну затримку, ефективне використання трафіку та незалежність безпеки від транспортного протоколу (TCP, UDP, HTTP, WebSockets). Хмарне зберігання дозволяє синхронізувати дані між пристроями без P2P у звичайних чатах. Розподілені дата-центри (близько 5 основних DC + CDN) гарантують низьку латентність і відмовостійкість. Мікросервіси дозволяють горизонтальне масштабування при мільярдах повідомлень на день. Такий підхід поєднує швидкість (легший за Signal Protocol), масштабованість (обробка мільйонів одночасних з’єднань) і безпеку (сервер-клієнтне шифрування + опціональне E2E).

3.3 Основні компоненти системи

Система складається з чітко відокремлених підсистем, взаємодія яких забезпечує надійність і продуктивність.
• Інтерфейс користувача (Client Layer): Мобільні (iOS/Android), десктопні та веб-клієнти. Використовують бібліотеку TDLib (відкрита, крос-платформена) для роботи з MTProto. Підтримують офлайн-режим, локальне кешування та миттєву синхронізацію.
• Серверна частина (MTProto Gateway / Front-end Servers): Кілька глобальних дата-центрів (DC1–DC5+), кожен з яких може працювати незалежно. MTProto Gateway приймає зашифровані з’єднання від клієнтів, виконує автентифікацію та передає запити далі.
• Рівень бізнес-логіки (Business Logic Layer):
o Message Router — роутер повідомлень, який визначає одержувача (користувач, група, канал) і доставляє їх.
o Group/Channel Engine — обробка великих груп (до 200 000 учасників) і каналів.
o Bot Platform — Bot API, обробка команд, платежів і webhook’ів.
o Media Processing — стиснення фото/відео, обробка стікерів і файлів.
• Рівень зберігання даних (Data Storage Layer): Cloud Storage — розподілене сховище повідомлень і медіа по дата-центрах. Використовується шарінг і реплікація. Медіа кешується через CDN. Дані для звичайних чатів зберігаються на серверах (хмарні), для секретних — лише на пристроях.
• Допоміжні сервіси (Auxiliary Services):
o Authentication Gateway — автентифікація та створення auth_key.
o Push Service — доставка push-сповіщень (FCM, APNs, власні).
o Search Service — глобальний пошук повідомлень, користувачів, «людей поруч».
o CDN Network — мережа edge-серверів для швидкої доставки медіа.
o Load Balancer — розподіл навантаження між DC.

3.4 Взаємодія компонентів

Потоки даних:
1. Клієнт формує повідомлення → серіалізує в TL-схему → шифрує за MTProto 2.0 (AES-256 IGE) → відправляє через транспорт (TCP/UDP/HTTP).
2. MTProto Gateway розшифровує (для cloud-чату), перевіряє seq_no і msg_id.
3. Message Router визначає маршрут → зберігає в Cloud Storage → передає Push Service для сповіщення одержувача.
4. Для груп/каналів залучається Group/Channel Engine.
5. Медіа завантажується через Media Processing → зберігається в CDN.
Протоколи взаємодії: MTProto 2.0 (RPC-стиль), внутрішні сервіси спілкуються через високошвидкісні черги подій. Синхронізація — eventual consistency між DC з реплікацією.
Механізми синхронізації:
• msg_id і seq_no для порядку та захисту від replay.
• Server salt (змінюється кожні 30 хв).
• ACK (msgs_ack) для підтвердження доставки.
• Multi-DC routing: клієнт підключається до найближчого DC, дані реплікуються.

3.5 Масштабованість та надійність

• Механізми балансування навантаження: Глобальні load balancer’и перед MTProto Gateway. Клієнт автоматично вибирає найближчий DC за latency.
• Відмовостійкість: Multi-DC архітектура — кожен дата-центр працює незалежно. При відмові одного DC трафік автоматично перенаправляється. Дані реплікуються між DC.
• Кешування: Memcache для часто використовуваних даних, CDN для медіа (фото/відео завантажуються з найближчого edge-сервера).
• Реплікація: Повна реплікація повідомлень і метаданих між дата-центрами. Медіа — географічно розподілене.
• Контейнеризація та хмарна інфраструктура: Cloud-native підхід з мікросервісами та database sharding. Використовуються контейнери (ймовірно Kubernetes/Docker) для горизонтального масштабування. Система витримує пікові навантаження (мільярди повідомлень на день) завдяки ефективному MTProto, який легший за традиційні WebSocket/HТТP.
Завдяки цьому Telegram підтримує майже необмежене масштабування без втрати швидкості.

3.6 Приклади програмної реалізації

Запити до сервісів ШІ: Telegram активно інтегрується з ШІ через Bot API. Наприклад, бот на базі Grok, GPT чи власних моделей Telegram AI може обробляти запити користувача. Запит відправляється через HTTPS (Bot API) або MTProto (для нативних клієнтів). Приклад: користувач пише боту @GrokBot — повідомлення йде через Message Router → Bot Platform → зовнішній ШІ-сервіс (webhook або API call) → відповідь повертається тим самим шляхом.


Фрагменти коду / API, що демонструють архітектурні принципи:
1. Приклад Bot API (HTTP, демонструє простоту бізнес-логіки):
import requests
TOKEN = "YOUR_BOT_TOKEN"
def send_message(chat_id, text):
    url = f"https://api.telegram.org/bot{TOKEN}/sendMessage"
    payload = {"chat_id": chat_id, "text": text, "parse_mode": "HTML"}
    response = requests.post(url, json=payload)  # MTProto Gateway обробляє внутрішньо
    return response.json()
2. Приклад MTProto RPC (високорівневий API, TL-схема): Відправка повідомлення (спрощено, реальний код у TDLib):
messages.sendMessage#d1d34c26 flags:INT peer:InputPeer message:string ...
3. Приклад інтеграції з ШІ-сервісом (webhook + бізнес-логіка): Бот отримує update через webhook → перевіряє chat_id → викликає зовнішній AI API (наприклад, xAI Grok API) → формує відповідь і надсилає назад через sendMessage. Це демонструє подієво-орієнтовану архітектуру (event → router → external service → response).

Програмний код
1) Код перевірки на доцільність повіомлення користувача

import os
import json
from dotenv import load_dotenv
from openai import AsyncOpenAI          # Асинхронний клієнт
from aiogram import Bot, Dispatcher, types
from aiogram.filters import Command
from aiogram import F

load_dotenv()

# === Налаштування ===
TELEGRAM_TOKEN = os.getenv("TELEGRAM_TOKEN")
XAI_API_KEY = os.getenv("XAI_API_KEY")

bot = Bot(token=TELEGRAM_TOKEN)
dp = Dispatcher()

# xAI Grok клієнт
grok_client = AsyncOpenAI(
    api_key=XAI_API_KEY,
    base_url="https://api.x.ai/v1"
)

# Системний промпт для Grok (дуже важливий!)
SYSTEM_PROMPT = """
Ти — строгий, але доброзичливий модератор і редактор. 
Перевір повідомлення користувача за двома критеріями:

1. КОРЕКТНІСТЬ:
   - Орфографія та граматика українською/російською/англійською.
   - Фактична точність (якщо є факти).
   - Логічна послідовність і відсутність суперечностей.

2. ДОЦІЛЬНІСТЬ:
   - Чи релевантне повідомлення контексту чату?
   - Чи корисне/конструктивне?
   - Чи не містить спаму, токсичності, реклами, образ?
   - Чи доцільно надсилати саме зараз?

Поверни ТІЛЬКИ валідний JSON (без додаткового тексту!):
{
  "correctness_score": 0-10,
  "expediency_score": 0-10,
  "verdict": "OK" | "CORRECT" | "IMPROVE" | "BLOCK",
  "issues": ["короткий список проблем"],
  "suggestion": "пропозиція виправленого повідомлення або пояснення",
  "explanation": "коротке пояснення для користувача"
}
"""

@dp.message(F.text)  # Реагуємо на будь-яке текстове повідомлення
async def check_message(message: types.Message):
    user_text = message.text.strip()
    if not user_text or len(user_text) < 3:
        return

    try:
        # Запит до Grok API
        response = await grok_client.chat.completions.create(
            model="grok-4.20-reasoning",          # або grok-4.20-non-reasoning для швидкості
            messages=[
                {"role": "system", "content": SYSTEM_PROMPT},
                {"role": "user", "content": f"Перевір це повідомлення:\n\n{user_text}"}
            ],
            max_tokens=800,
            temperature=0.3,          # низька температура для стабільності
            response_format={"type": "json_object"}   # примушуємо JSON
        )

        grok_result = json.loads(response.choices[0].message.content)

        # Логіка реакції бота
        verdict = grok_result.get("verdict", "IMPROVE")
        suggestion = grok_result.get("suggestion", "")
        explanation = grok_result.get("explanation", "")

        if verdict == "OK":
            await message.reply("✅ Повідомлення коректне та доцільне!")
        elif verdict == "CORRECT":
            await message.reply(f"🔧 Виправлено:\n{suggestion}\n\n{explanation}")
        elif verdict == "IMPROVE":
            await message.reply(f"⚠️ Рекомендації:\n{explanation}\n\nПропозиція: {suggestion}")
        elif verdict == "BLOCK":
            await message.delete()
            await message.answer(f"🚫 Повідомлення заблоковано.\nПричина: {explanation}")

    except Exception as e:
